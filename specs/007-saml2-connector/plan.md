# Implementation Plan: SAML 2.0 Upstream Connector

**Branch**: `007-saml2-connector` | **Date**: 2026-04-19 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/007-saml2-connector/spec.md`

## Summary

Add a SAML 2.0 Service Provider (SP) role to netidm that allows enterprise users to authenticate via Okta, ADFS, Azure AD, or Shibboleth. Introduces a new `SamlClient` entry class (DL22 migration), a `SamlClientProvider` struct analogous to `OAuth2ClientProvider`, a new `handler_saml_client.rs` auth handler, two new HTTP endpoints (SSO redirect initiation + ACS), and CLI management commands. XML signature verification is handled by the `samael` 0.0.20 crate with the `xmlsec` feature.

## Technical Context

**Language/Version**: Rust stable (see `rust-toolchain.toml`)
**Primary Dependencies**: `samael` 0.0.20 with `xmlsec` feature (SAML 2.0 parsing + XML signature verification); all other deps already present (`axum`, `askama`, `compact_jwt`, `netidmd_lib`, `netidm_proto`)
**New Dependencies**: `samael = { version = "0.0.20", features = ["xmlsec"] }` in `server/lib/Cargo.toml` and `server/core/Cargo.toml`; requires native system packages `libxml2-dev`, `libxslt-dev`, `libxmlsec1-dev`, `pkg-config`
**Storage**: Netidm MVCC entry database ‚Äî DL22 migration adds new `EntryClass::SamlClient` with 9 attributes; DB nonce via existing `AssertionNonce` infrastructure
**Testing**: `cargo test` via `server/testkit` integration test infrastructure (real in-process netidmd)
**Target Platform**: Linux server (same as rest of netidm)
**Project Type**: Library + HTTP service + CLI tool
**Performance Goals**: ACS response processing adds one XML signature verification per login; acceptable for interactive login
**Constraints**: `server/lib` must remain network-free; in-memory AuthnRequest map lives in `server/core` state; samael's `xmlsec` feature requires C native deps at build time

### Layer 5: Auth Handler (`server/lib/src/idm/authsession/handler_saml_client.rs`)

New handler ‚Äî analogous to `handler_oauth2_client.rs`:

**`init_saml` step**:
1. Load `SamlClientProvider` by name
2. Generate `request_id = format!("_{}", Uuid::new_v4().simple())`
3. `samael::AuthnRequest::builder()` ‚Üí set issuer, ACS URL, name ID policy
4. Deflate-compress + base64-encode ‚Üí `saml_request`
5. `relay_state = Uuid::new_v4().to_string()`
6. Return `CredState::External(AuthExternal::SamlAuthnRequest { ... })`

**`validate_acs_response` step**:
1. Decode `saml_response` from base64
2. `samael::Response::try_from_xml()` (parse)
3. Check `InResponseTo` vs stored request in in-memory map (consume)
4. `samael::Response::verify_signature(&idp_cert)` (XML signature)
5. Extract `Assertion.Conditions` ‚Üí check NotBefore/NotOnOrAfter ¬±5min
6. Check nonce (`server/lib/src/server/assert.rs` ‚Äî `check_and_update_assertion_nonce()`)
7. Extract `NameID` ‚Üí `external_id`
8. Extract mapped attributes ‚Üí `email`, `display_name`, `group_names`
9. JIT provision or find existing account
10. Write nonce with `DeleteAfter = NotOnOrAfter`
11. Return `CredState::Success { auth_type: AuthType::SamlFederated, ... }`

### Layer 6: HTTP Endpoints (`server/core/src/https/views/login.rs`)

**`GET /ui/sso/:name`** (`view_saml_sso_get`):
- `handle_auth(InitSamlProvider { name })` ‚Üí expect `AuthExternal::SamlAuthnRequest`
- Store `(relay_state ‚Üí SamlAuthRequest)` in `ServerState.saml_pending_requests: DashMap<String, SamlAuthRequest>`
- Redirect: `Location: <sso_url>?SAMLRequest=<saml_request>&RelayState=<relay_state>`

**`POST /ui/login/saml/:name/acs`** (`view_saml_acs_post`):
- Parse form body: `SAMLResponse`, `RelayState`
- `handle_auth(SamlAcsResponse { saml_response, relay_state, provider_name: name.clone() })`
- On `CredState::Success` ‚Üí issue session cookie, redirect to original URL
- On `CredState::Denied` ‚Üí render login error page

Both routes added to `unguarded_csp_router` (no HTMX guard ‚Äî IdP posts directly).

**Login page (`/ui/login`)**: Template already receives provider list from `LoginView`. Add SAML providers to the provider list alongside OAuth2 providers so SSO buttons appear.

## Complexity Tracking

No constitution violations. This feature introduces one new entry class (analogous to `OAuth2Client`) and one new C-native dependency (`samael`/`xmlsec`). The native dependency is unavoidable ‚Äî no pure-Rust XML digital signature library exists. All other patterns (DL migration, ACP, handler, CLI) directly mirror the existing OAuth2 client connector.
