# Implementation Plan: Social Login with JIT Provisioning

**Branch**: `001-social-login-jit` | **Date**: 2026-04-16 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/001-social-login-jit/spec.md`

## Summary

Add GitHub and Google as social login providers to Kanidm with Just-In-Time (JIT) account provisioning. When a user authenticates via a social provider for the first time, Kanidm automatically creates their account from the provider's identity claims, presents a confirmation page for username review, and then logs them in — all within a single flow. Returning users are recognised by a stable provider subject identifier and logged into their existing account.

The existing `OAuth2ClientProvider` + `CredHandlerOAuth2Client` + `OAuth2AccountCredential` machinery handles the PKCE redirect flow; this feature extends it with claim extraction, a new `CredState::ProvisioningRequired` variant, a JIT provisioning write transaction, and two new UI routes.

## Technical Context

**Language/Version**: Rust (stable, current toolchain — check `rust-toolchain.toml`)
**Primary Dependencies**: Internal Kanidm crates; `reqwest` for userinfo HTTP calls; `jsonwebtoken`/existing JWT handling for Google `id_token` verification; `serde_json` for claim parsing
**Storage**: Kanidm's internal B+ tree database (QueryServer) — no external DB. New DL15 schema migration adds 5 optional attributes.
**Testing**: `cargo test` — all tests must pass on a clean checkout. Integration tests in `server/lib/src/` test module.
**Target Platform**: Linux server (existing deployment target)
**Project Type**: Server daemon / identity provider — extending existing auth flow
**Performance Goals**: Social login round-trip must complete within existing session timeout (~5 min). Userinfo endpoint call (GitHub path) must not block the auth flow for more than 5 seconds (use reqwest timeout).
**Constraints**: `cargo test` must work without external services. JIT provisioning write transaction must be atomic — partial account creation must not persist.
**Scale/Scope**: Per-provider configuration; number of JIT-provisioned accounts is bounded only by the provider's user base.

Create `server/lib/src/migration_data/dl15/schema.rs`:
- Add `SchemaAttribute` definitions for each new attribute (syntax: Url for endpoint, Boolean for jit flag, Utf8String for claim maps)
- Extend `SCHEMA_CLASS_OAUTH2_CLIENT.systemmay` to include all 5 new attributes
- Wire into the migration level system (follow DL14 pattern)

### Step 2 — OAuth2ClientProvider Struct + Loading

**File**: `server/lib/src/idm/oauth2_client.rs`

Add to `OAuth2ClientProvider`:
```rust
pub(crate) userinfo_endpoint: Option<Url>,
```

Update `reload_oauth2_client_providers()` to extract:

- `jit_provisioning` via `get_ava_single_bool(Attribute::OAuth2JitProvisioning)` → `bool` (default false if absent)
- `claim_map`: extract each of the 3 claim map attributes into a `HashMap<KanidmAttr, String>`

### Step 3 — ExternalUserClaims + Claim Extraction

**File**: `server/lib/src/idm/authsession/handler_oauth2_client.rs`

Add `ExternalUserClaims` struct (see data-model.md).

Add to `CredHandlerOAuth2Client`:
```rust
userinfo_endpoint: Option<Url>,
jit_provisioning: bool,
claim_map: HashMap<KanidmAttr, String>,
```

Modify `validate_access_token_response()` to:
1. If `response.id_token.is_some()` → decode JWT, extract OIDC claims (Google path)
2. Else if `self.userinfo_endpoint.is_some()` → call userinfo endpoint with `Bearer <access_token>`, parse JSON (GitHub path)
3. Else → existing behaviour (no claims, return `CredState::Success` as before for providers without JIT)
4. Build `ExternalUserClaims` from extracted data
5. If `self.jit_provisioning` and claims available → return `CredState::ProvisioningRequired { provider_uuid: self.provider_id, claims }`
6. If no existing account found (checked in Step 4) → route to provisioning

Note: Account lookup (Step 4) must be called before deciding between `Success` and `ProvisioningRequired`. This requires passing a query capability into the handler — use a callback or pass the lookup result in from `authsession/mod.rs`.

---

### Step 4 — Account Lookup

**File**: `server/lib/src/idm/server.rs`

Add to `IdmServerProxyReadTransaction` (or write transaction — whichever handles auth):
```rust
pub fn find_account_by_oauth2_provider_and_user_id(
    &mut self,
    provider_uuid: Uuid,
    sub: &str,
) -> Result<Option<Account>, OperationError>
```

Filter:
```rust
filter_all!(f_and!([
    f_eq(Attribute::OAuth2AccountProvider, PartialValue::Refer(provider_uuid)),
    f_eq(Attribute::OAuth2AccountUniqueUserId, PartialValue::Utf8(sub.to_string())),
]))
```

Returns `None` if no match, `Some(Account)` if found.

---

### Step 5 — JIT Provisioning Function

**File**: `server/lib/src/idm/server.rs`

Add to `IdmServerWriteTransaction`:
```rust
pub fn jit_provision_oauth2_account(
    &mut self,
    provider_uuid: Uuid,
    claims: &ExternalUserClaims,
    desired_name: &str,
) -> Result<Uuid, OperationError>
```

Logic:
1. Verify `desired_name` is a valid iname and not taken (return `OperationError::InvalidAttribute` if invalid, `OperationError::ValueExists` if taken)
2. Derive `display_name`: `claims.display_name.clone().unwrap_or_else(|| desired_name.to_string())`
3. Generate `cred_id = Uuid::new_v4()`
4. Build entry with classes: `Object, Account, Person, OAuth2Account`
5. Set attributes: `name`, `displayName`, `mail` (if present), `oauth2AccountProvider`, `oauth2AccountUniqueUserId`, `oauth2AccountCredentialUuid`
6. Call `self.qs_write.internal_create(vec![entry])` — use existing pattern from other `create_*` functions
7. Return new account UUID

Edge cases:
- `claims.sub` empty → `OperationError::InvalidAttribute`
- `(provider_uuid, sub)` already exists → `OperationError::ValueExists` (guard against race)
- `desired_name` collision after lookup → `OperationError::ValueExists` (caller retries with suggestion)

### Step 6 — AuthSession: ProvisioningRequired State

**File**: `server/lib/src/idm/authsession/mod.rs`

1. Add `CredState::ProvisioningRequired { provider_uuid: Uuid, claims: ExternalUserClaims }` variant
2. Add `AuthState::ProvisioningRequired { provider_uuid: Uuid, claims: ExternalUserClaims }` variant
3. In auth credential processing: when handler returns `CredState::ProvisioningRequired`, propagate to `AuthState::ProvisioningRequired`
4. The `issue_uat()` function MUST NOT be callable from `ProvisioningRequired` state — enforce via match arms

### Step 7 — UI: Confirmation Page + Routes

**File**: `server/core/src/https/views/login.rs`

1. Add `COOKIE_OAUTH2_PROVISION_REQ` constant
2. Modify `view_login_oauth2_landing()`: on `AuthState::ProvisioningRequired`, serialize claims to signed cookie, redirect to `/ui/login/provision`
3. Add `view_login_provision_get()`: read cookie, derive proposed username (call username derivation logic), render Askama template
4. Add `view_login_provision_post()`: read cookie, validate submitted username, call `jit_provision_oauth2_account()`, on success call `issue_uat()` and redirect
5. Register both routes in the Axum router (follow existing route registration pattern)
6. Create Askama template `server/core/src/https/templates/login_provision.html`

- `SetIdentityClaimMap`: validate `kanidm_attr` ∈ {name, displayname, email}, write appropriate `oauth2_claim_map_*` attribute
