## v0.1.33 — TOML preload system and WireGuard connect improvements

### TOML preload system (replaces hjson migrations)

Netidm now reads structured TOML files from a `preload.d/` directory on startup and on
`SIGHUP` reload. This replaces the hjson-based SCIM migration system entirely.

**How it works**

Files must match `\d\d-name.toml` and are applied in lexicographic order. Each file is
SHA-256 hashed; if the hash is unchanged since the last apply the file is skipped — making
every apply idempotent regardless of how many times the server starts or reloads.

Each resource kind is declared with a TOML array table:

```toml
[[tunnel]]
[[user]]
[[group]]
[[oauth2_client]]
[[connector]]
```

A single file may contain any mix of resource kinds.

**Supported resource types**

| Table | What it asserts |
|---|---|
| `[[tunnel]]` | WireGuard tunnel (interface, endpoint, address pool, keys) |
| `[[user]]` | Person account with optional SSH keys and inline WireGuard peers |
| `[[group]]` | Regular or POSIX group with member list |
| `[[oauth2_client]]` | OAuth2 / OIDC relying party with scope maps |
| `[[connector]]` | Upstream identity connector (GitHub, OIDC, LDAP, SAML, …) |

**Resource identity**

UUIDs are derived deterministically from the resource kind and name using
`UUID v5(NAMESPACE_OID, "netidm:preload:{kind}:{name}")`. The same name always produces the
same UUID, so re-applying a file asserting a resource that was already created by an earlier
run will locate and update it rather than creating a duplicate.

**Removing a file**

Removing a file from `preload.d/` stops future assertions but does **not** delete the
resource from the database. Preload is assert-only.

**Configuration**

The directory is configured in `server.toml`:

```toml
preload_path = "/etc/netidm/preload.d"
```

Default paths by platform:

| Platform | Default |
|---|---|
| Linux (release) | `/etc/netidm/preload.d` |
| Debian package | `/etc/netidm/preload.d` |
| FreeBSD | `/usr/local/etc/netidm/preload.d` |
| Container | `/data/preload.d` |
| Developer | `/tmp/netidm/preload.d` |

**Migration from hjson**

The `migration_path` config field and the hjson SCIM migration system have been removed.
The `serde-hjson` dependency has been dropped from the workspace. If you had
`migration_path` set in `server.toml`, replace it with `preload_path` pointing to a
directory of TOML files. See the updated [Entry Management](book/src/entry_management.md)
chapter for the full TOML schema.

### WireGuard connect — re-sync path and PSK support

The `POST /wg/connect` endpoint now handles two cases:

**Re-sync (existing peer)**

If the authenticated caller already has a peer with the same public key registered on the
requested tunnel, the server returns the stored configuration and PSK without consuming the
registration token. This allows devices to re-run the connect script after a reinstall or
key wipe and recover their WireGuard config in a single call. The response includes
`"status": "existing"`.

**New registration**

On first registration the flow is unchanged: the token is validated and consumed, an IP
address is allocated from the tunnel pool, and a WgPeer entry is created. The response
includes `"status": "new"`.

**PSK support**

Pre-shared keys are now a first-class part of the connect flow:

- The client may supply a PSK in the `preshared_key` field of the request. If omitted, the
  server generates a 32-byte cryptographically random PSK.
- The PSK is stored on the WgPeer entry and returned in the response as `psk`.
- On re-sync the stored PSK is returned so the client can restore its configuration exactly.
- The generated `wg-quick` config block now includes a `PresharedKey = …` line in the
  `[Peer]` section.

**Hostname-based peer naming**

The request now accepts an optional `hostname` field. When provided, the peer entry is named
`peer-<sanitised-hostname>-<caller-uuid>` for easier identification in the database and
audit logs. When omitted, the previous format `peer-<caller-uuid>-<tunnel-name>` is used.

**Config block fix**

The `MTU` line is now emitted in the `[Interface]` section (where wg-quick expects it)
rather than as a comment in the `[Peer]` section. This was a bug in the previous config
generation path.

**API changes**

`WgConnectRequest`:

```json
{
  "token": "...",
  "pubkey": "...",
  "preshared_key": "...",   // optional — omit to auto-generate
  "hostname": "my-laptop"   // optional — used for peer entry name
}
```

`WgConnectResponse`:

```json
{
  "config": "...",
  "address": ["10.64.0.2/32"],
  "server_pubkey": "...",
  "endpoint": "wg.example.com:51820",
  "psk": "...",             // PSK to embed in the client config
  "status": "new"           // "new" or "existing"
}
```

Existing clients that do not send `preshared_key` or `hostname` continue to work — both
fields are optional with `null` treated as absent.

---

### No schema migration required

This release does not add a new domain level. Existing databases continue to work without
any operator action beyond the `migration_path` → `preload_path` rename if you had that
field set.
