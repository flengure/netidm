## v0.1.30 — Full dex feature parity

This release closes the final six gaps in the dex feature parity roadmap. Netidm now replicates
every runtime capability of dex that is relevant to production deployments. No further dex-parity
work is planned.

### G1 — Full `grant_types_supported` in OIDC discovery

The `/.well-known/openid-configuration` discovery document now advertises all grant types the
server supports: `authorization_code`, `refresh_token`, `urn:ietf:params:oauth:grant-type:device_code`,
and `password`. Clients that inspect discovery to decide which flows to attempt (e.g.
`oauth2-proxy`, Vault, many OIDC libraries) no longer need manual configuration overrides.

### G2 — Resource Owner Password Credentials grant (RFC 6749 §4.3)

The token endpoint now accepts `grant_type=password` requests. The credential is validated
through the connector named in `[expiry] password_connector` — any connector that implements
`authenticate_password` (currently LDAP) can serve as the password backend. Scope validation
checks the requesting client's `client_scopes`; per-group scope maps are not evaluated (adequate
for M2M and legacy CLI flows). The grant is advertised in discovery.

### G3 — CORS wiring from config

CORS handling for OIDC/token endpoints is now driven by `[web] allowed_origins` in the server
config. The field accepts a list of origin strings; `["*"]` allows all origins. Previously, CORS
headers were added per-handler with no config knob. The new implementation uses
`tower_http::CorsLayer` applied at the router level, consistent with dex's gorilla/cors
integration.

### G4 — Token expiry config wired to `TokenPolicy`

`[expiry]` config section fields are now fully wired into the runtime `TokenPolicy`:

- `access_token_lifetime` — lifetime of issued access tokens (default: 1 hour)
- `refresh_token_lifetime` — absolute maximum lifetime of a refresh token session (default: 30 days)
- `refresh_token_expiry` — sliding window after last use before a refresh token expires
- `password_connector` — name of the connector used for ROPC password validation (G2)

### G5 — Device authorization flow promoted to stable

The OAuth 2.0 Device Authorization Grant (RFC 8628) is now unconditionally available. The
`device_flow` feature gate has been removed. The flow endpoints
(`/oauth2/device_authorization`, `/oauth2/device`) are always registered. The grant type is
advertised in discovery (G1).

### G6 — Cross-client audience scope (`trustedPeers`)

Resource servers can now declare a list of trusted peer clients. A trusted peer may request
`audience:server:client_id:<peer_name>` as a scope, which causes the issued token's `aud` claim
to include the peer's client ID. This is the netidm equivalent of dex's `staticClients[*].trustedPeers`.

The `aud` claim in RFC 9068 structured tokens is now serialized as a bare string for single-audience
tokens and a JSON array for multi-audience tokens, consistent with RFC 7519 §4.1.3 and existing
RP parsers.
