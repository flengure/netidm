## v0.1.32 — Integration test coverage for ROPC and cross-client audience

This release adds end-to-end integration tests for two features that shipped in v0.1.30 but had
no HTTP-level test coverage: the Resource Owner Password Credentials grant (G2) and the
cross-client audience scope / trusted peers (G6).

### G2 — ROPC rejection verified end-to-end

A new integration test exercises the full HTTP path for `grant_type=password` and confirms that
the server returns `400 Bad Request` when no `password_connector` is configured. This is the
enforcement boundary documented in the v0.1.30 release: the ROPC grant is inert unless
`oauth2.password_connector` is explicitly set in server config.

The positive path (successful ROPC against a real LDAP connector) requires a live LDAP fixture
not yet present in the test harness; a tracking note has been left in the test file.

### G6 — Cross-client audience verified end-to-end

A new integration test exercises the complete auth-code flow for a client requesting an
`audience:server:client_id:<peer>` scope:

1. Two resource servers are created (`grants_test_rs_a`, `grants_test_rs_b`).
2. RS_B's `trusted_peers` list is set to include RS_A via the internal modify API.
3. RS_A runs a PKCE auth-code flow requesting scope
   `audience:server:client_id:grants_test_rs_b`.
4. The issued JWT's `aud` claim is decoded and asserted to be a JSON array containing
   both RS_A and RS_B — confirming the `OneOrMany<_, PreferOne>` serialisation emits an
   array for multi-audience tokens (RFC 7519 §4.1.3).
5. RS_A introspects its own token to confirm it remains active.

The test also documents the introspection boundary: RS_B cannot call netidm's
`/oauth2/token/introspect` for a token signed by RS_A, because each RS's introspect endpoint
verifies against its own key object. External services accepting a cross-client token verify it
directly against the issuing RS's JWKS endpoint — the same pattern dex's `trustedPeers` uses.

### No operator action required

This release contains no configuration changes, schema migrations, or API surface changes.
Existing deployments need no action.
