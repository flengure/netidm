## v0.1.31 — Prometheus metrics endpoint

Netidm now exposes a Prometheus-compatible `/metrics` endpoint and ships an OTLP metrics
pipeline alongside the existing OTLP trace pipeline.

### Prometheus scrape endpoint

`GET /metrics` returns the full metric set in Prometheus text format (version 0.0.4). The
endpoint is unauthenticated by design — scrape access should be controlled at the network/proxy
layer. Content-Type is `text/plain; version=0.0.4; charset=utf-8`.

The following metrics are exported:

| Metric | Type | Labels | Description |
|---|---|---|---|
| `netidm_build` | gauge | `version` | Always 1; used to read the running version from Prometheus |
| `netidm_http_requests_total` | counter | `method`, `path`, `status` | Total HTTP requests by method, matched route template, and status code |
| `netidm_http_request_duration_seconds` | histogram | `method`, `path` | Request latency in seconds (11 buckets: 5 ms – 10 s) |
| `netidm_forward_auth_decisions_total` | counter | `outcome` | Forward-auth gate decisions: `allow`, `deny`, `skip` |
| `netidm_oauth2_token_grants_total` | counter | `grant_type`, `outcome` | Token endpoint results by grant type and `success`/`error` |
| `netidm_auth_decisions_total` | counter | `outcome` | IDM `/v1/auth` outcomes: `success`, `denied`, `error` |
| `netidm_oauth2_authorize_decisions_total` | counter | `outcome` | OAuth2 authorize endpoint outcomes: `consent_pending`, `auto_permitted`, `auth_required`, `denied`, `error` |
| `netidm_oauth2_consent_decisions_total` | counter | `outcome` | Consent screen outcomes: `permitted`, `rejected`, `error` |

HTTP path labels use the matched route template (e.g. `/oauth2/:rs_name/token`) rather than the
raw URI, preventing high-cardinality label explosion from path parameters.

### OTLP metrics pipeline

When `otel_grpc_url` is configured, an OTLP metrics exporter is started alongside the existing
trace exporter. Metrics are pushed every 30 seconds to the same collector endpoint using the same
`OTEL_EXPORTER_OTLP_HEADERS` authorization header. The provider is shut down cleanly on process
exit so in-flight data is flushed.

The metrics exported via OTLP are independent of the Prometheus endpoint — both can be active
simultaneously, or either can be used alone.

### Implementation notes

- Metrics are held in a process-wide singleton (`LazyLock<Arc<NetidmMetrics>>`) so all subsystems
  can record without threading metrics through function arguments.
- The HTTP middleware uses axum's `MatchedPath` extension (available via `route_layer`) so path
  templates are captured, not raw URLs.
- The `netidm_build` gauge is recorded at startup with the running version string, enabling
  Prometheus alerting on unexpected version changes across a fleet.
